## 2024-12-16 - Prevent PHP Object Injection in Pluggable Queues via Strict Deserialization
**Vulnerability:** Pluggable queue systems (such as `database`, `redis`, `beanstalkd`, `sync`) deserialize jobs stored as payload strings. Because PHP's `unserialize()` was called without any restriction on `allowed_classes`, any malicious payload injected into the queues could trigger PHP Object Injection (POP gadget chain), allowing arbitrary code execution or arbitrary file deletion on the worker server.
**Learning:** In a decentralized queue or RPC architecture where jobs or messages are serialized and passed around, trusting the content of raw payloads for unrestricted `unserialize()` is extremely risky. Since every job payload specifies its expected `job` class name, we can strictly restrict `unserialize()`'s `allowed_classes` option to exactly the expected class name.
**Prevention:** Never use unrestricted `unserialize($data)`. Always pass a strict whitelist of classes using `['allowed_classes' => [$jobClass]]` or `['allowed_classes' => false]` to prevent unexpected object instantiation and magic methods from triggering.

## 2025-01-14 - Prevent Username/Email Enumeration via Login Timing Side-Channels and CPU DoS on Large Passwords
**Vulnerability:** The authentication system verified passwords using `password_verify` only if the user was found in the database. When the user was not found, the function was bypassed entirely. Because BCrypt password verification is computationally expensive, this timing difference allowed attackers to scan and enumerate valid emails. Furthermore, extremely long password inputs were not capped, introducing a CPU exhaustion Denial-of-Service (DoS) vector during password hashing.
**Learning:** Even if queries use safe prepared statements, execution flow differences between user-exist and user-absent branches can leak secret states via high-resolution timing measurements. In addition, security-related CPU-heavy operations must always be protected with maximum size boundaries to avoid denial of service.
**Prevention:** Always verify a pre-computed dummy BCrypt hash when the query result is empty, ensuring both flows run the hashing function. Enforce a strict length limit (such as 255 characters) on all user password inputs before invoking hashing or verification APIs.
